# Domain Model and Ontology

**Version:** 0.1  
**Status:** Draft

## 1. Purpose

This document defines the core domain vocabulary, object identities, relationships, lifecycle states, provenance, and ontology-governance rules.

## 2. Core aggregates

### Source
A configured origin of information. Fields include `id`, `type`, `name`, `owner_id`, `configuration_ref`, `enabled`, `default_project_id`, and timestamps.

### SourceObject
A stable logical object in a source system, such as a file, issue, pull request, commit, message, or manually entered note.

### SourceObjectVersion
An immutable observed version with source revision, content hash, observed time, MIME type, extraction status, and content reference.

### ContentChunk
A bounded derived segment with structural position, text, token count, embedding-model identifier, vector, and source-version reference.

### Entity
A stable real-world or conceptual identity. Initial entity types are Project, Person, Organization, Document, Repository, Technology, Concept, SystemComponent, and ExternalSystem.

### Claim
A structured assertion connecting a subject to a predicate and value or object. A claim includes temporal validity, confidence, status, ontology version, and provenance.

### Event
A time-bounded occurrence, such as a meeting, commit, release, or status change.

### Decision
A governed conclusion with rationale, status, effective date, affected entities, and evidence.

### Task
An actionable item with owner, status, priority, due date, project, dependencies, and evidence.

### KnowledgeProposal
A non-canonical suggested change generated by a processor or agent.

### OntologyProposal
A governed suggestion to change entity types, relationship types, fields, aliases, or validation rules.

## 3. Stable identity

Every domain object uses an opaque UUID. Source identity is represented separately through `(source_id, external_id)`. Content hashes identify immutable versions but MUST NOT replace logical IDs.

## 4. Core relationships

Examples include:

- `BELONGS_TO_PROJECT`
- `MENTIONS`
- `AUTHORED_BY`
- `PARTICIPATED_IN`
- `CREATED_FROM`
- `SUPPORTED_BY`
- `CONTRADICTS`
- `SUPERSEDES`
- `DEPENDS_ON`
- `AFFECTS`
- `IMPLEMENTS`
- `USES_TECHNOLOGY`
- `ASSIGNED_TO`
- `DECIDED_IN`

Relationships are typed, directed where meaningful, and may carry temporal and provenance properties.

## 5. Claim lifecycle

`proposed → approved → active → superseded | rejected | retracted`

No approved claim is physically overwritten in place. Corrections create a new version or superseding claim and retain the audit trail.

## 6. Provenance

Each claim and derived relationship MUST reference one or more evidence records containing source object version, chunk or span, extraction method, processing run, model, prompt, schema, confidence, and approval decision.

## 7. Temporal model

The system distinguishes:

- transaction time: when the system recorded the information;
- valid time: when the information applies in the real world;
- observation time: when the source was observed.

## 8. Ontology governance

The normative ontology is stored in version-controlled YAML. AI can propose but cannot directly publish ontology changes. New types should be introduced only when existing types and aliases cannot represent the concept without loss.

## 9. Validation rules

- A canonical object must satisfy its schema.
- A claim must have provenance.
- A relationship type must be allowed for the source and target types.
- Deprecated aliases may be read but are not emitted for new data.
- Unknown types remain proposals until approved.
